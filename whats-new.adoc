== 2026 年 5 月 3 日



=== 适用于 NetApp ONTAP 的 Amazon FSX



==== 块设备克隆

Workload Factory 为您的块设备提供克隆，以便您可以快速配置可写入且高效的数据副本，同时控制容量、访问权限和生命周期。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/manage-block-storage.html["在 NetApp Workload Factory 中管理块存储"]



==== 块设备上的 EC2 标签

块设备表现在包括一列，该列显示使用每个块设备的 EC2 主机的标签。这有助于您在故障排除或审核期间确定哪个主机使用每个块设备。

要在每个块设备上创建标记，请使用我们提供的用户脚本部署 EC2 实例。

link:https://github.com/NetApp/FSx-ONTAP-samples-scripts/tree/main/Management-Utilities/ec2-user-data-iscsi-create-and-mount["用于为块设备分配标签的 EC2 用户数据脚本"]



==== 适用于 ONTAP 文件系统的 FSx 容量趋势

直接从 FSx for ONTAP 文件系统的概述中了解 SSD 容量趋势。新的容量管理卡可全面了解上个月的使用趋势。该分析包括实际使用的容量和分配的容量之间的比较，以便对资源利用率进行明确评估。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/manage-capacity-automatically.html["在 NetApp Workload Factory 中自动管理容量"]



=== 设置和管理



==== Workload Factory 的电子邮件通知

现在，您可以通过电子邮件接收 Workload Factory 通知，而无需配置 Amazon SNS 主题。启用电子邮件通知，以将通知发送到与帐户关联的任何电子邮件地址。

您可以从 Workload Factory 控制台或 NetApp Console 配置电子邮件通知。NetApp Console 还提供配置出现在铃铛图标下的 NetApp Console 通知的选项。

如果在 NetApp Console 中配置通知，NetApp Console 将拥有通知传递的所有权。Workload Factory 控制台中的通知设置变为只读，以防止重复通知。



==== EDA 延迟新通知

NetApp Workload Factory 通知服务包括 EDA 延迟通知，当 EDA 操作的延迟超过特定阈值时通知您。

link:https://docs.netapp.com/us-en/workload-setup-admin/configure-notifications.html["配置 Workload Factory 通知"]



=== EDA 工作负载



==== 延迟事件的电子邮件和 SNS 通知

现在，当在 FSx for ONTAP 卷上检测到延迟事件时，您可以接收自动通知。在 Workload Factory 设置中配置电子邮件或 Amazon SNS 通知，以便在卷违反配置的延迟阈值时接收实时警报。

通知按文件系统发送，包括有关所有受影响卷、严重性级别、阈值比较的详细信息，以及直接链接到 Latency 页面进行调查。

https://docs.netapp.com/us-en/workload-eda/task-monitor-latency.html#configure-latency-notifications["了解如何配置延迟通知"^]。



==== 使用交互式图形实现延迟趋势可视化

延迟分析面板现在包括一个选项卡式界面，用于全面的延迟分析：

* *概述*：查看自动基本分析结果，显示哪个组件导致延迟
* *随着时间的推移*：使用跨多个时间范围（1 小时、3 小时、12 小时、24 小时或 72 小时）的交互式图表可视化卷延迟行为


*随着时间的推移*选项卡显示原始 CloudWatch 延迟数据（读取或写入，具体取决于触发的警报），以及警告和关键阈值线、违规指标和每个违规行为的详细指标。

这种全面的可视化可帮助您识别模式，了解延迟问题是重复出现还是孤立存在，并就补救措施做出更明智的决策。

https://docs.netapp.com/us-en/workload-eda/task-analyze-latency-trends.html["了解如何分析延迟趋势"^]。



== 2026 年 4 月 6 日



=== 适用于 NetApp ONTAP 的 Amazon FSX



==== 精心设计的分析更新

Workload Factory 分析您的 FSx for ONTAP 文件系统中的*非活动 NAS 卷*。配置分析可识别未主动使用的 NAS 卷，并建议采取措施来优化存储利用率和降低成本。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/improve-configurations.html["为 ONTAP 文件系统实施架构良好的 FSx"]



==== S3 接入点的日志表功能

日志表功能提供了一种大规模管理 FSx ONTAP S3 对象存储的方法，允许您轻松审核存储桶中对象的用户访问事件和操作。使用 Workload Factory，您可以设置存储桶和 S3 表的自动扫描，以保留事件记录，显示所有对象、其详细信息和用户操作。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/setup-journal-table.html["设置 NetApp Workload Factory 的日记表基础结构"]



==== 支持管理现有的 igroups

Workload Factory 支持管理现有的 igroups。您可以添加或删除块设备和主机启动器，以便按时交付部署的工作负载。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/manage-block-storage.html["在 NetApp Workload Factory 中管理块存储"]



==== 支持为 FlexVol 卷指定聚合

Workload Factory 支持在选定的聚合上创建 FlexVol 卷。这允许您在所有文件系统资源之间分配容量和性能负载。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/create-volume.html["在 NetApp Workload Factory 中创建 FlexVol volume"]



==== 块设备的上次访问时间

Workload Factory 报告块设备的最后访问时间。时间基于您上次在 Workload Factory 中访问 igroup 的时间。通过了解每个块设备的最后使用时间，您可以优化资源。



=== 数据库工作负载



==== 针对 Oracle 的良好架构分析

精心设计的分析包括以下针对 Oracle 的新配置：

* *Oracle Critical Patch Updates*：评估受支持的 Oracle 自我管理数据库安装缺少的安全补丁。
* *跨区域复制*：检查是否为服务于 Oracle 数据库的 FSx for ONTAP 文件系统启用了跨区域复制 (CRR)。
* *应用程序一致性快照*：检查 Oracle 数据库是否受到 NetApp SnapCenter 应用程序一致性快照的保护。


link:https://docs.netapp.com/us-en/workload-databases/optimize-configurations.html["在 Workload Factory 中实施架构良好的数据库配置"]



==== Oracle 一次性评估

Workload Factory 现在提供一次性评估，以便在需要时分析 Oracle 数据库部署的配置 - 新部署、迁移后、故障排除等。您将获得建议，以帮助您优化部署配置并遵守最佳实践。这是您在不完全致力于设置过程的情况下使用精心设计的功能的机会。您可以评估数据库，而无需在 Workload Factory 中存储凭据、注册数据库资源或设置外部连接。

link:https://docs.netapp.com/us-en/workload-databases/optimize-configurations.html["对 Oracle 数据库部署执行一次性评估"]



==== 改进了多个 Oracle 数据库的成本节约

我们增强了 Oracle 的成本节约分析，在一个 FSx for ONTAP 文件系统上为多个 Oracle 本地数据库提供整合的存储解决方案。这种整合通过减少多个数据库所需的文件系统数量来优化存储成本，从而节省成本。

link:https://docs.netapp.com/us-en/workload-databases/explore-savings.html#explore-savings-for-detected-hosts["探索在本地 Oracle 上检测到的数据库的节省"]

=== EDA 工作负载



==== EDA 容量建议

Workload Factory 现在会自动分析 EDA 库存中每个 FSx for ONTAP 集群的 SSD 容量，并在需要调整时生成容量建议。该算法为每个集群每 24 小时运行一次。

确定建议后：

* 您会立即收到电子邮件或 WAD 通知（不包含在每周摘要中）。
* 在 Clusters 表中，受影响的集群上出现一个灯泡指示器。
* 概览控制面板上的摘要显示了有活动建议的集群总数。


选择具有活动建议的集群以查看建议，该建议解释了建议的更改和推理。

https://docs.netapp.com/us-en/workload-eda/task-use-overview-dashboard.html#capacity-recommendations["详细了解容量建议"^]。



==== 自动 AI 延迟分析

延迟分析现在包括 AI 代理分析。

对于涉及数据或集群延迟的复杂场景，您可以调用 AI-agent 分析以接收：

* 详细的根本原因识别
* 受影响的 EC2 客户端列表
* 特定修复步骤


AI 代理分析需要 Amazon Bedrock 模型 ARN 配置，并且在基本分析建议进行更深入调查的情况下可按需提供。AI 代理可以识别欺凌卷、非最佳挂载点配置、FlexGroup 重新平衡需求和横向扩展要求等问题。

https://docs.netapp.com/us-en/workload-eda/task-monitor-latency.html#run-ai-agent-analysis["详细了解 AI 代理分析"^]。

=== VMware工作负载



==== 隆重推出针对 Google Cloud VMware Engine 的迁移顾问支持

NetApp Workload Factory 现在支持 Google Cloud VMware Engine。您可以使用 Google Cloud NetApp Volumes 作为外部 NFS 数据存储将本地 VMware 工作负载迁移到 Google Cloud VMware Engine，从而优化成本并实现对 VMware 环境的更大控制，而无需重构或重新平台化应用程序。

Google Cloud VMware Engine 迁移顾问提供：

* 支持 VE1 和 VE2 节点类型的自动化 Google Cloud VMware Engine 集群规模调整和设计
* vSAN 内部数据存储区和 Google Cloud NetApp Volumes 外部 NFS 数据存储区之间的虚拟机布局优化。
* 仅 vSAN 和 Google Cloud NetApp Volumes 增强型配置之间的详细成本比较和 TCO 分析。
* 支持标准、高级和 Extreme Google Cloud NetApp Volumes 服务级别。
* 使用 Terraform 进行基础设施即代码部署，以实现自动化配置。

=== 数据库工作负载



==== SQL Server 一次性评估

Workload Factory 现在提供一次性评估，以便在需要时分析 SQL Server 数据库部署的配置 - 新部署、迁移后、故障排除等。您将获得建议，以帮助您优化部署配置并遵守最佳实践。这是您在不完全致力于设置过程的情况下使用精心设计的功能的机会。您可以评估数据库，而无需在 Workload Factory 中存储凭据、注册数据库资源或设置外部连接。

link:https://docs.netapp.com/us-en/workload-databases/optimize-configurations.html["对数据库部署执行一次性评估"]



==== Microsoft SQL Server 上 Always On 可用性组和 Oracle 上 Data Guard 的注册支持

现在，您可以在 NetApp Workload Factory 的数据库工作负载中，使用 Always On 可用性组 (AOAG) 部署模型发现和注册 Microsoft SQL Server 实例，并使用 Data Guard 部署模型注册 Oracle 数据库。此增强功能允许在 Workload Factory 控制台中更好地管理 SQL Server 和 Oracle 资源。

注册 AOAG 实例和 Data Guard 数据库后，您可以利用 Workload Factory for Databases 中的各种功能，例如查看资源详细信息和分析错误日志。

Workload Factory for Databases 还支持使用 AOAG 部署模型对 Microsoft SQL Server 实例进行精心设计的分析。您可以分析可用性组中 SQL Server 实例的配置，并接收优化部署配置和遵守最佳实践的建议。注册您的 AOAG 实例以启用此功能。

link:https://docs.netapp.com/us-en/workload-databases/register-instance.html#register-a-sql-server-instance["在 Workload Factory for Databases 中注册资源"]



==== 针对 Oracle 的良好架构分析

精心设计的分析包括 Oracle 的*操作系统补丁*，用于检查您是否在操作系统上安装了最新的补丁，并告诉您如何安装它们。

此外，当存储布局等配置依赖于其他配置时，Oracle 的精心设计的分析会通知您。该通知告诉您要优化链接配置的操作，以便我们提供的所有建议都是最佳的。

link:https://docs.netapp.com/us-en/workload-databases/optimize-configurations.html["在 Workload Factory 中实施架构良好的数据库配置"]



==== Oracle 本地数据库环境成本分析和迁移规划

Workload Factory for Databases 现在可以分析并帮助您规划 Oracle 本地数据库迁移到 Amazon FSx for NetApp ONTAP。您可以使用节省计算器估算在云中运行 Oracle 本地数据库环境的成本，并查看将本地数据库环境迁移到云的建议。

link:https://docs.netapp.com/us-en/workload-databases/explore-savings.html["探索本地数据库环境的节约"]

==== 支持 Storage VM 迁移

NetApp Workload Factory 现在支持存储虚拟机的迁移。此功能允许将 ONTAP 存储系统数据和配置从本地 ONTAP 系统或第一代 FSx for ONTAP 文件系统迁移到第二代 FSx for ONTAP 文件系统。您可以复制存储虚拟机数据和配置设置以移动到新的文件系统，同时将停机时间和对用户和应用程序的中断降至最低。

要使用此功能，link:https://docs.netapp.com/us-en/workload-fsx-ontap/create-replication.html["创建复制关系"] 并选择 *Migration* 作为用例。要完成迁移过程，您必须 link:https://docs.netapp.com/us-en/workload-fsx-ontap/create-replication.html#cut-over-replication-for-migration-use-cases["切换 Storage VM 及其复制卷"] 立即将数据和存储 VM 配置设置永久迁移到目标 FSx for ONTAP 文件系统。
